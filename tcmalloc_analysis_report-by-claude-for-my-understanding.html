1. Thread Cache vs TCMALLOC_MAX Limit

The chart below shows actual thread cache usage after each workload type, plotted against the configured limit (red dashed line = limit). Points above the red line indicate the thread cache is exceeding its configured budget.

Key Observations

Natural Equilibrium: Thread cache usage converges to a workload-determined equilibrium regardless of the limit setting. For large (24M) objects: ~30–76 MiB. For medium (2M): ~20–50 MiB. For small (64K): ~8–32 MiB. Setting TCMALLOC_MAX above this equilibrium wastes the budget. Setting it below causes overcrowding and inter-thread stealing pressure.
⚠️ Small Objects are Deceptive: Small (64K) object workloads show the highest saturation percentages at low limits (8/16 MiB) because small objects generate many more allocations per operation, quickly filling per-thread free-lists. However, their absolute memory footprint (10–32 MiB) is the lowest of the three sizes.
⚠️ Mixed Workloads are the Worst Case: Mixed (write + read + delete) workloads consistently produce 1.5–2× more thread cache growth than pure read or write. The combination of allocation patterns from three operation types saturates thread caches fastest.

3. Page Heap (Global Pool) — The Real Memory Hoarder

The page heap is TCMalloc's global free-list. When threads free memory and their per-thread caches are full, freed spans go to the page heap rather than back to the OS. This is the primary driver of RSS growth.

Page Heap Dynamics

🔴 Large Objects Create Massive Page Heap Accumulation: Writing 50 × 24 MiB objects causes the page heap to jump from ~90–100 MiB to 500–540 MiB — an increase of 410–445 MiB. This memory stays in the page heap indefinitely (0 bytes released to OS in all tests). This is the dominant source of Ceph RGW's memory footprint growth under write-heavy workloads.
Write → Accumulate, Read → Reuse: Write workloads push memory into the page heap. Subsequent read workloads pull from the page heap (slight decrease), but the drain is slow — a 540 MiB page heap loses only ~2–10 MiB during 50 read operations. The page heap is effectively a one-way valve under write-heavy conditions.
⚠️ Medium Objects (2M): Consistent ~100 MiB Page Heap Accumulation: Regardless of cache setting, writing 50 × 2 MiB objects creates approximately 100 MiB of page heap. This suggests TCMalloc retains roughly the equivalent of 1–2 full working sets in its page heap.
✅ Small Objects are Efficient: Writing 50 × 64K objects only creates 3–8 MiB in the page heap. Small object memory is efficiently recycled via the central and transfer caches before falling to the page heap.

Central Cache & Transfer Cache

These two caches sit between the thread caches and the page heap. Central cache is a global free-list per size class. Transfer cache shuttles objects between central cache and thread caches.

For all workloads, central cache stays 1–5 MiB and transfer cache stays 1–8 MiB — very stable. These are not the memory consumers to watch. The thread cache and page heap dominate entirely.

4. Memory Component Heatmap (Write-Only Workloads)

Each cell shows the MiB used by each allocator component after writing 50 objects, for each object size and cache setting.

App In-Use is constant: The application layer uses ~120 MiB regardless of object size or cache setting. The RGW's actual working set doesn't grow — only the allocator's bookkeeping and free-list overhead scales with object size.
Metadata overhead scales with spans: The number of spans in use grows from ~640 (baseline) to 2000+ after heavy mixed workloads for small objects. Each span is a contiguous run of TCMalloc pages. More spans = more metadata = more memory for the allocator's internal accounting.

5. Total Actual Memory by Cache Setting & Workload

Write workloads always dominate RSS: Write-only total actual memory is consistently higher than read or mixed for large objects. For large (24M) write-only: ~730–750 MiB total; for read-only: ~680–730 MiB; for mixed: ~680–740 MiB.
⚠️ Cache setting barely affects total RSS: For large objects, the total actual memory varies only ~30 MiB across the entire 8→512 MiB range of cache settings. The page heap dominates, and the page heap size is driven by workload pattern and object size — not by the TCMALLOC_MAX setting.

6. Thread Heaps In Use — Parallelism & Contention

What Thread Heaps Tell Us

Each active thread gets its own heap. More thread heaps = more parallelism = more per-thread cache overhead, but also better lock-free performance.

Large & medium objects: 598 thread heaps (saturated). This is the apparent ceiling for this Ceph configuration. All threads are active, indicating the workload is fully parallel for larger objects.
Small objects: 237–548 thread heaps. Write-only workloads use fewer threads (~240) than read-only (~480) or mixed (~540). Small writes are CPU-bound and may be serialized in RGW's encoding pipeline, reducing parallelism.

Per-Thread Cache Budget

With 598 threads and TCMALLOC_MAX = 32 MiB, each thread gets an average budget of just 54 KB. With 128 MiB, it's 213 KB. With 512 MiB, it's 853 KB.

7. Actionable Recommendations

A. Optimal TCMALLOC_MAX_TOTAL_THREAD_CACHE_BYTES Setting

Set to 128 MiB (134217728 bytes) for general-purpose Ceph RGW workloads.
— Comfortably above natural equilibrium for all tested object sizes
— Provides ~213 KB per thread at 598 thread heaps — enough for efficient caching
— Going higher (256/512 MiB) provides zero measurable improvement as thread caches never approach those limits
— Lower settings (≤32 MiB) risk inter-thread stealing overhead, especially for large objects

B. Address Page Heap Bloat — The Real RSS Driver

The page heap is the dominant source of RSS growth — not the thread cache. To control it:
  • Call MallocExtension::instance()->ReleaseFreeMemory() periodically in RGW to flush the page heap back to the OS
  • Or set TCMALLOC_RELEASE_RATE > 0 (default 0) to enable background OS release — even a value of 1.0 will help
  • Consider tcmalloc_release_rate = 5.0 in ceph.conf if using Ceph's TCMalloc integration

C. Write-Heavy Workloads Need Special Attention

For deployments handling predominantly large objects (>16 MiB) with heavy write traffic: the page heap will accumulate 400–500 MiB per burst of 50 large writes. This is not a leak — it's TCMalloc holding freed pages for reuse. But if you're seeing RSS grow monotonically, enabling TCMALLOC_RELEASE_RATE is the fix, not increasing TCMALLOC_MAX.

D. Central & Transfer Cache — No Action Needed

Central cache (1–5 MiB) and transfer cache (1–8 MiB) are working correctly and efficiently. These stay within normal bounds across all 63 experiments. No tuning needed.
